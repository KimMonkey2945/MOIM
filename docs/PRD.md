# 모임 이벤트 관리 웹 MVP — PRD

> **제품명:** Moim (모임)
> **한 줄 요약:** 운동·친목·스터디 등 모든 소모임 주최자가 공지·참여·카풀·정산을 한 화면에서 처리하는 원스톱 이벤트 관리 웹
> **작성일:** 2026-06-16
> **버전:** v1.0
> **작성자:** 솔로 개발자

---

## 1. 문제 정의 & 배경

### 현재 상황

소모임 주최자는 하나의 이벤트를 운영하기 위해 여러 채널을 동시에 오가야 한다.

| 업무              | 현재 도구               | 문제점                        |
| ----------------- | ----------------------- | ----------------------------- |
| 공지 및 일정 안내 | 카카오톡 단톡방         | 메시지 매몰, 링크 유실        |
| 참여 의사 수집    | 네이버 폼 / 댓글        | 수동 집계, 실시간 파악 불가   |
| 카풀 조율         | 단톡방 + 엑셀           | 좌석 중복, 연락 반복          |
| 정산              | 더치트리 / 스프레드시트 | 입력 이중화, 송금 확인 수작업 |

### 핵심 문제

- **정보 분산**: 공지·참여자·카풀·비용이 플랫폼 3~4개에 흩어짐
- **주최자 번아웃**: 반복적인 수동 집계와 개별 연락 업무
- **참여자 불편**: "내가 참석 신청을 했는지" "카풀 배정이 됐는지" 확인 불가

## 2. 목표 & 비목표

### 목표 (v1 MVP)

- [ ] 주최자가 이벤트 하나를 5분 안에 생성하고 참여자를 초대할 수 있다
- [ ] 참여자가 RSVP(참석/불참/미정)를 원클릭으로 변경할 수 있다
- [ ] 공지와 댓글이 단일 피드로 관리된다
- [ ] 카풀 좌석 등록 → 탑승 신청 → 주최자 승인 플로우가 완결된다
- [ ] 비용을 입력하고 N분의1 / 차등 분배로 각자 금액을 확인할 수 있다
- [ ] **주최자 대시보드** 한 화면에서 참여 현황·미승인 카풀·미정산을 동시에 파악한다

### 비목표 (Non-goals, v1 제외)

- 게스트(비회원) 링크 초대
- 반복 일정(Recurring Event)
- 이메일/푸시 알림
- 토스·카카오페이 직접 송금 연동
- 캘린더 외부 동기(Google Calendar 등)
- 멤버십 요금제 / 광고
- 모임 그룹(상위 조직) 관리

## 4. 핵심 사용자 시나리오 (유저 스토리)

### 주최자 시나리오

```
As a 주최자,
I want to 이벤트를 생성하고 참여자들의 RSVP를 한 화면에서 보고 싶다
So that 참석 여부를 일일이 물어보지 않아도 된다.

As a 주최자,
I want to 카풀 신청자를 승인/거절하고 좌석 배정 현황을 볼 수 있다
So that 단톡방에서 혼선 없이 카풀을 정리할 수 있다.

As a 주최자,
I want to 비용을 입력하고 분배 방식(균등/차등)을 선택하면 각자 금액이 자동 계산된다
So that 정산 스프레드시트를 따로 만들 필요가 없다.

As a 주최자,
I want to 대시보드에서 이벤트별 미완료 항목(미승인 카풀, 미정산)을 한눈에 본다
So that 놓치는 항목이 없다.
```

### 참여자 시나리오

```
As a 참여자,
I want to 이벤트 상세 페이지에서 참석/불참/미정을 클릭 한 번으로 변경한다
So that 주최자에게 따로 메시지를 보낼 필요가 없다.

As a 참여자,
I want to 카풀 운전자로 등록하거나 탑승 신청을 할 수 있다
So that 카풀 조율을 앱 안에서 해결한다.

As a 참여자,
I want to 내가 내야 할 금액과 송금 완료 여부를 확인한다
So that 정산이 끝났는지 주최자에게 다시 묻지 않아도 된다.
```

## 6. 화면 / IA 구조

### 6.1 라우트 트리

```
/
├── (public)
│   ├── /                        # 랜딩 / 홈 피드 (로그인 후 리디렉션)
│   └── /auth
│       ├── /login               # 로그인
│       ├── /sign-up             # 회원가입
│       ├── /confirm             # 이메일 OTP 확인 (Route Handler)
│       ├── /forgot-password     # 비밀번호 찾기
│       └── /update-password     # 비밀번호 재설정
│
└── /protected
    ├── /                        # 홈 피드 (이벤트 카드 목록)
    ├── /dashboard               # 주최자 대시보드
    ├── /events
    │   ├── /new                 # 이벤트 생성
    │   └── /[eventId]           # 이벤트 상세
    │       ├── /edit            # 이벤트 수정 (주최자)
    │       ├── /announcements   # 공지 탭
    │       │   └── /[annoId]    # 공지 상세 + 댓글
    │       ├── /carpools        # 카풀 탭
    │       └── /expenses        # 정산 탭
    └── /profile                 # 내 프로필 수정
```

### 6.2 주요 화면 목록

| 화면명          | 라우트                   | 주요 구성요소                                                                    |
| --------------- | ------------------------ | -------------------------------------------------------------------------------- |
| 홈 피드         | `/protected`             | 검색바, 카테고리 칩, 이벤트 카드 그리드, FAB(이벤트 생성)                        |
| 주최자 대시보드 | `/protected/dashboard`   | RSVP 위젯, 미승인 카풀 위젯, 미정산 위젯, 이벤트 목록 위젯                       |

| 이벤트 상세     | `/protected/events/[id]` | 히어로 배너, 이벤트 정보, RSVP 버튼, 탭 내비(공지/카풀/정산), 참여자 아바타 스택 |

| 회원가입        | `/auth/sign-up`          | 이름·이메일·PW 폼, Google 버튼                                                   |
| 프로필          | `/protected/profile`     | 닉네임·아바타 수정 폼                                                            |

## 7. UI/UX 디자인 가이드

> Figma "Travel App" 템플릿 디자인 언어를 참조한 방향성이다. 실제 컴포넌트는 shadcn/ui(new-york) 기반으로 커스터마이징한다.

### 7.1 레이아웃 원칙

- **모바일 퍼스트**: 375px 기준 설계 → 768px(tablet) → 1280px(desktop) 반응형
- **카드 중심**: 이벤트·공지·카풀 모두 `rounded-2xl` 카드로 표현
- **넉넉한 여백**: 섹션 간 `gap-6` 이상, 카드 내부 `p-5`
- **하단 탭 (모바일)**: 고정 `h-16` 탭바, 활성 탭 강조색

### 7.2 타이포그래피

| 용도        | 클래스 예시                         |
| ----------- | ----------------------------------- |
| 페이지 제목 | `text-3xl font-bold tracking-tight` |
| 섹션 제목   | `text-xl font-semibold`             |
| 카드 제목   | `text-base font-semibold`           |
| 보조 텍스트 | `text-sm text-muted-foreground`     |

### 7.3 컬러 & 테마

- **강조색(Primary)**: 브랜드 컬러 1개 (CSS 변수 `--primary` 커스터마이징)
- **표면색**: `card`, `popover` shadcn 토큰 활용
- **다크모드**: `next-themes`의 `class` 전략, 모든 색상 토큰은 CSS 변수로

### 7.4 이벤트 카드 구성요소

```
┌──────────────────────────────┐
│  [썸네일 이미지 16:9]        │
│  카테고리 칩 (운동/친목...)  │
│  이벤트 제목 (2줄 말줄임)    │
│  📅 날짜·시간  📍 장소       │
│  [아바타 스택] +N명 참석     │
│  [RSVP 상태 배지]            │
└──────────────────────────────┘
```

### 7.5 주요 인터랙션

- **RSVP 버튼**: 3-segment 컨트롤 (참석 ✓ / 미정 ? / 불참 ✗), 선택 시 즉시 반영
- **카풀 카드**: 운전자 이름 + 잔여 좌석 게이지 바 + "탑승 신청" 버튼
- **정산 금액**: 내 납부액은 큰 볼드 폰트로 상단 강조, 전체 내역은 하단 테이블
- **대시보드 위젯**: 숫자 KPI 카드 (큰 숫자 + 레이블 + 화살표 링크)
- **로딩**: Skeleton 카드 (shadcn `Skeleton` 컴포넌트)

## 8. 데이터 모델 개요

> 상세 컬럼 정의는 Supabase 마이그레이션 파일에서 관리한다. 여기서는 엔티티·관계 요약만 기술한다.

### 8.1 주요 엔티티

```
users (Supabase auth.users 기본 제공)
  └─< profiles          1:1   사용자 공개 프로필 (username, display_name, avatar_url)

profiles
  └─< events            1:N   주최자가 생성한 이벤트

events
  ├─< event_participants 1:N  참여자별 RSVP 상태
  ├─< announcements     1:N   이벤트 내 공지
  ├─< carpools          1:N   카풀 운전자 등록
  └─< expenses          1:N   비용 항목

announcements
  └─< comments          1:N   댓글 스레드

carpools
  └─< carpool_requests  1:N   탑승 신청 (신청자, 상태: pending/approved/rejected)

expenses
  └─< expense_shares    1:N   참여자별 분담 금액 + 송금 완료 여부
```

### 8.2 핵심 필드 요약

| 테이블               | 주요 컬럼                                                                                                                                           |
| -------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------- |
| `profiles`           | `id(FK→auth.users)`, `display_name`, `avatar_url`, `updated_at`                                                                                     |
| `events`             | `id`, `host_id(FK→profiles)`, `title`, `description`, `category`, `event_at`, `location`, `thumbnail_url`, `status(active/cancelled)`, `created_at` |
| `event_participants` | `id`, `event_id`, `user_id`, `rsvp_status(going/maybe/not_going)`, `joined_at`                                                                      |
| `announcements`      | `id`, `event_id`, `author_id`, `content`, `is_pinned`, `created_at`                                                                                 |
| `comments`           | `id`, `announcement_id`, `author_id`, `content`, `created_at`                                                                                       |
| `carpools`           | `id`, `event_id`, `driver_id`, `total_seats`, `departure_note`, `status(open/full/closed)`                                                          |
| `carpool_requests`   | `id`, `carpool_id`, `requester_id`, `status(pending/approved/rejected)`, `created_at`                                                               |
| `expenses`           | `id`, `event_id`, `title`, `total_amount`, `paid_by(FK→profiles)`, `split_type(equal/custom)`, `created_at`                                         |
| `expense_shares`     | `id`, `expense_id`, `user_id`, `amount`, `is_paid`, `paid_at`                                                                                       |

## 9. 권한 / 보안 (RLS 방향)

> 모든 테이블에 Supabase Row Level Security(RLS)를 활성화한다. 인증 미확인 요청은 전부 차단.

| 테이블               | SELECT             | INSERT         | UPDATE            | DELETE          |
| -------------------- | ------------------ | -------------- | ----------------- | --------------- |
| `profiles`           | 로그인 사용자 전체 | 본인           | 본인              | —               |
| `events`             | 참여자 + 주최자    | 로그인 사용자  | 주최자            | 주최자          |
| `event_participants` | 이벤트 참여자 전체 | 본인           | 본인(RSVP)        | 본인            |
| `announcements`      | 이벤트 참여자 전체 | 주최자         | 주최자            | 주최자          |
| `comments`           | 이벤트 참여자 전체 | 이벤트 참여자  | —                 | 본인 + 주최자   |
| `carpools`           | 이벤트 참여자 전체 | 이벤트 참여자  | 운전자 + 주최자   | 운전자 + 주최자 |
| `carpool_requests`   | 운전자 + 신청자    | 이벤트 참여자  | 운전자(승인/거절) | 신청자(취소)    |
| `expenses`           | 이벤트 참여자 전체 | 주최자         | 주최자            | 주최자          |
| `expense_shares`     | 본인 + 주최자      | 주최자(시스템) | 주최자(송금확인)  | —               |

**구현 주의사항**

- 이벤트 접근은 `event_participants`에 레코드가 있거나 `host_id`가 본인인 경우로 제한
- 이벤트 생성 시 주최자 본인을 `event_participants`에 자동 INSERT (DB 트리거 또는 Server Action)
- 쓰기 작업은 Server Actions에서 처리하고, 클라이언트에서는 조회 전용 쿼리만 사용
- 인증 사용자 확인은 `supabase.auth.getClaims()` 사용 (프로젝트 컨벤션, `getUser()` 아님)

## 11. 성공 지표 (MVP KPI)

| 지표               | 목표                                     | 측정 방법                                     |
| ------------------ | ---------------------------------------- | --------------------------------------------- |
| 이벤트 생성 완료율 | 생성 시작 대비 80% 이상 완료             | 생성 폼 진입 vs. 저장 완료 이벤트 수          |
| RSVP 응답률        | 이벤트 참여자 중 70% 이상 RSVP 입력      | `event_participants.rsvp_status != null` 비율 |
| 카풀 처리 완료율   | 신청 건 중 90% 이상 승인/거절 처리       | `carpool_requests.status != pending` 비율     |
| 정산 완료율        | 이벤트 종료 후 7일 내 `is_paid` 80% 달성 | `expense_shares.is_paid` 비율                 |
| 주최자 재방문율    | 1개월 내 2번째 이벤트 생성 50%           | 사용자별 이벤트 생성 이력                     |
